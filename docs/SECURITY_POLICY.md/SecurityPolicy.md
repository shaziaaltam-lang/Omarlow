# سياسة الأمن في مشروع Omarlow

تلتزم منصة Omarlow بأعلى معايير الحماية الرقمية لضمان سلامة بيانات المستخدمين والمحامين والوثائق القانونية. يوضح هذا المستند البروتوكولات الأمنية المعتمدة في النظام.

## 1. تشفير البيانات (Data Encryption)
- **بيانات الراحة (At Rest)**: يتم تشفير كافة البيانات الحساسة في قاعدة البيانات باستخدام خوارزمية AES-256.
- **أثناء الانتقال (In Transit)**: يتم فرض بروتوكول TLS 1.3 لجميع الاتصالات عبر HTTPS لضمان عدم اعتراض البيانات.
- **إدارة المفاتيح**: يتم تخزين مفاتيح التشفير بشكل آمن خارج نطاق الكود المصدري باستخدام متغيرات البيئة المحمية (Environment Variables) وتطبيق مبدأ الفصل بين البيئات.

## 2. سياسة التحقق الثنائي (Two-Factor Authentication - 2FA)
- **التفعيل**: يُعد التحقق الثنائي خط دفاع أول لجميع الحسابات ذات الصلاحيات الإدارية.
- **الآلية**: يعتمد النظام على رموز OTP المؤقتة (Time-based One-Time Password) عبر التطبيقات المعتمدة.
- **الاسترداد**: يتم توفير رموز استرداد مشفرة (Recovery Codes) تُستخدم في حالات فقدان الوصول لجهاز التوثيق.

## 3. حماية الواجهات والاتصالات (API Security)
- **التحجيم (Rate Limiting)**: تم تطبيق قيود صارمة على طلبات الواجهات (API Rate Limiting) لمنع هجمات حجب الخدمة (DDoS) والقوة الغاشمة (Brute Force).
- **رؤوس الأمان (Security Headers)**: يتم إرسال رؤوس أمان متقدمة (مثل Content-Security-Policy, X-Frame-Options) عبر Middleware مخصص لضمان حماية المتصفحات.

## 4. الاستجابة للحوادث الأمنية (Incident Response)
في حالة اكتشاف خرق أمني، يلتزم فريق العمل بالخطوات التالية:
1. **الاحتواء**: عزل النظام أو الخدمة المتأثرة فوراً.
2. **التحليل**: فحص سجلات التدقيق (Audit Logs) وتحديد نطاق الاختراق.
3. **المعالجة**: سد الثغرة الأمنية وتحديث التصحيحات (Patches).
4. **التبليغ**: إخطار الأطراف المتأثرة وفقاً للوائح الخصوصية المعمول بها.

## 5. سجلات التدقيق (Audit Logs)
يحتفظ النظام بسجلات دقيقة لكل العمليات الحساسة (الولوج، تعديل الوثائق، تغيير الصلاحيات) عبر `AuditLogMiddleware` و `ActivityLogService` لضمان التتبع والمحاسبة.
